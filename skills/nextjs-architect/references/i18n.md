# Internationalization (i18n)

Use **next-intl** for App Router — the standard library for Next.js i18n. ~2KB, native
Server Component support, TypeScript-safe, ICU message format.

## Architecture

```
messages/
  en.json             # { "common": {...}, "home": {...}, "product": {...} }
  ru.json
  zh.json
src/
  i18n/
    request.ts        # Request-scoped locale config
    routing.ts        # Locale routing definitions
  middleware.ts       # Locale detection (Next.js 15) or proxy.ts (Next.js 16+)
  app/
    [locale]/         # Dynamic locale segment
      layout.tsx      # NextIntlClientProvider
      page.tsx
```

## Server vs Client Components

```tsx
// Server Component — getTranslations (async, zero client JS)
import { getTranslations } from 'next-intl/server'

export default async function HomePage() {
  const t = await getTranslations('home')
  return <h1>{t('title')}</h1>
}

// Client Component — useTranslations hook
'use client'
import { useTranslations } from 'next-intl'

export function AddToCartButton() {
  const t = useTranslations('product')
  return <button>{t('addToCart')}</button>
}
```

## Translation Format (ICU)

```json
{
  "items": "{count, plural, =0 {No items} one {# item} other {# items}}",
  "greeting": "Hello, {name}!",
  "lastSeen": "Last seen {date, date, medium}"
}
```

## TypeScript Safety

```ts
// global.d.ts
import en from '../messages/en.json'
type Messages = typeof en
declare global {
  interface IntlMessages extends Messages {}
}
```

## Rules
- URL strategy: `[locale]` path prefix (`/en/about`, `/ru/about`).
- Server Components use `getTranslations()` — zero client JS for static text.
- Use ICU format for plurals, dates, numbers — not manual string interpolation.
- RTL: set `dir` attribute on `<html>` based on locale. Use CSS logical properties
  (`margin-inline-start`, not `margin-left`).
- Translation files: JSON, namespaced by feature/page.
- If i18n not present in project but user creates content-facing pages, suggest setup.
